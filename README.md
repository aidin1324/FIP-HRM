### Обзор Проекта

ZernoHub – это интегрированная платформа CRM & HRM, предназначенная для управления ресторанными бизнес-процессами. Она объединяет функциональности управления человеческими ресурсами и взаимоотношениями с клиентами в единой системе, предоставляя удобные, мощные и полезные инструменты для эффективного управления рестораном.

### Структура Проекта

- **Landing Page**
    - Статический сайт, отображающий основные возможности проекта и служащий точкой входа для пользователей.

- **HRM-system-ZERNO**
    - **Backend:**
        - **Emps:**
            - Управление обратной связью (feedback) и связанными с ней функциями.
        - **Users:**
            - Аутентификация, регистрация, управление профилями и другие связанные функции.
        - **База Данных:**
            - Используется PostgreSQL с разделением на различные схемы для хранения данных, что позволяет экономить ресурсы и поддерживать логическую независимость модулей.
    
    - **Frontend: Evaluate**
        - Фронтенд-часть, предназначенная для оценки и создания обратной связи от гостей.
     - **Frontend: admin-panel-module**
        - Фронтенд-часть, предназначенная для CRM & HRM фичей для менеджеров и сотрудников.

### Технологии

- **Фронтенд:**
    - **React:** Используется для разработки пользовательского интерфейса.
    - **npm:** Управление зависимостями и сборка проекта.

- **Бэкэнд:**
    - **FastAPI:** Разработка серверной части приложения.
    - **Poetry:** Управление зависимостями и виртуальными окружениями для Python.

- **Контейнеризация:**
    - Каждый модуль содержит свой Docker-файл.
    - Все Docker-контейнеры собираются с помощью Docker Compose, формируя общую сеть для взаимодействия модулей.
    - Для запуска проекта достаточно выполнить команду `docker-compose up`, что автоматически настроит и запустит всю необходимую инфраструктуру.

### Тестирование

- **Backend:**
    - Используется технология **Locust** для проведения нагрузочного тестирования.
    - Тестовые сценарии расположены в файле `test`, обеспечивая проверку производительности и стабильности серверной части.

ZernoHub — это интегрированная CRM (Customer Relationship Management) и HRM (Human Resource Management) платформа, разработанная специально для ресторанной индустрии. Главная цель проекта — предоставить единый инструмент для управления бизнес-процессами ресторана: от взаимодействия с клиентами (опросы, обратная связь, программы лояльности) до учёта и развития персонала (роли, KPI, оценки, eNPS).

### 1.1 Основная Идея и Цели

1. **Универсальность**  
   Объединить все ключевые процессы и данные ресторана в одной системе — от регистрации новых сотрудников до сбора отзывов от клиентов.

2. **Эффективность**  
   Сократить количество разрозненных сервисов для управления бизнесом, тем самым снизив сложность, затраты и время принятия решений.

3. **Гибкость**  
   Использовать модульную архитектуру, позволяющую легко дорабатывать и масштабировать функциональность по мере роста ресторана или изменения его потребностей.

### 1.2 Особенности

- **Поддержка eNPS**  
  Позволяет оценивать удовлетворённость сотрудников и их вовлечённость.
- **Сбор и управление отзывами (Feedback)**  
  Клиенты могут быстро оставлять обратную связь, а администраторы — анализировать эти данные и принимать оперативные решения.
- **Аутентификация и роли**  
  Предусмотрены различные уровни доступа для разных групп пользователей (администратор, менеджер, сотрудник, гость).

## 2. Общая Структура Проекта

Проект ZernoHub разделён на несколько крупных частей, каждая из которых отвечает за свой функциональный блок:

1. **Landing Page**  
   Статический сайт, презентующий ключевые преимущества ZernoHub и служащий точкой входа для новых пользователей.

2. **HRM-system-ZERNO**  
   - **Backend** (FastAPI + PostgreSQL)  
     - **Emps**  
       Отвечает за сбор, хранение и управление отзывами (feedback), рейтингами и метриками eNPS.  
     - **Users**  
       Обеспечивает сервисы аутентификации (регистрация, логин), управление профилями и другими пользовательскими функциями.  
     - **База данных**  
       Единая PostgreSQL-база с разделением сущностей по схемам (schemas) для упрощения логической структуры и предотвращения конфликтов данных.
   - **Frontend: Evaluate**  
     Приложение для гостей ресторана, позволяющее оставлять и просматривать отзывы (comments, рейтинги).
   - **Frontend: admin-panel-module**  
     Интерфейс для менеджеров и администраторов (CRM & HRM): просмотр статистики, контроль KPI, управление отзывами и профилями сотрудников.

### 2.1 Почему PostgreSQL + Разные Схемы?

#### Использование одной базы данных с различными схемами

В современных информационных системах часто возникает необходимость организации и хранения большого объема данных. Один из популярных способов решения этой задачи — использование реляционных баз данных, таких как PostgreSQL. При этом важным моментом является выбор архитектурного подхода к разделению данных. Один из таких подходов — использование одной базы данных с несколькими схемами. В этом README мы рассмотрим, почему данный подход является выгодным, а также его преимущества и недостатки, а также влияние на масштабирование.

Использование одной базы данных с различными схемами позволяет структурировать данные внутри единой СУБД, при этом избегая необходимости создания отдельных баз данных для каждого типа данных или модуля системы. Схемы в PostgreSQL — это логические контейнеры для объектов базы данных (таблиц, представлений, индексов), которые могут быть использованы для разграничения данных между различными частями системы. Это подход упрощает администрирование и позволяет эффективно использовать ресурсы системы.

### Преимущества использования разных схем

1. **Централизация данных**  
   Все данные хранятся в одной базе данных, что упрощает администрирование и позволяет избежать сложностей синхронизации данных между несколькими базами данных. Также это упрощает процесс резервного копирования и восстановления данных, так как все данные находятся в одном месте.

2. **Изоляция данных**  
   Разделение данных по схемам позволяет изолировать различные области данных внутри одной базы. Например, данные клиентов могут быть хранены в одной схеме, а данные сотрудников — в другой. Это повышает безопасность и предотвращает случайное изменение данных, относящихся к различным частям системы.

3. **Оптимизация производительности**  
   Использование одной базы данных с несколькими схемами позволяет избежать дублирования данных и ускоряет выполнение запросов. Это также упрощает обработку данных, поскольку доступ к данным можно организовать через одну точку — базу данных, не переходя между несколькими источниками данных.

4. **Управление доступом**  
   Схемы позволяют настроить различные уровни доступа для пользователей, что может быть полезно в многопользовательских системах. Администраторы могут предоставлять доступ только к определённым схемам, ограничивая доступ к чувствительной информации.

5. **Гибкость и модульность**  
   Разделение на схемы позволяет разработать систему таким образом, чтобы различные части приложения (например, для работы с клиентами, сотрудниками, заказами) могли развиваться независимо, без риска вмешательства в данные других модулей.

### Недостатки использования различных схем

1. **Усложнение запросов**  
   Когда данные находятся в разных схемах, может возникнуть необходимость в более сложных запросах, особенно если требуется объединить информацию из нескольких схем. Хотя это не является значительным недостатком, иногда это может увеличить сложность работы с данными.

2. **Ограничения на взаимные связи**  
   В PostgreSQL существует возможность создавать связи между таблицами, находящимися в разных схемах. Однако такие операции могут быть менее эффективными, чем в случае, если все данные хранятся в одной схеме, что может снизить производительность при работе с большими объемами данных.

3. **Требования к администрированию**  
   Несмотря на то, что схема упрощает управление данными в пределах одной базы, в некоторых случаях требуется тщательное планирование архитектуры и дополнительных настроек для обеспечения изоляции и безопасности. Например, необходимо правильно настроить роли и права доступа, чтобы ограничить доступ к данным, хранящимся в других схемах.

## Масштабирование с использованием схем

Использование одной базы данных с несколькими схемами имеет несколько преимуществ с точки зрения масштабирования. Когда система растёт и появляется необходимость в добавлении новых функциональных модулей или увеличении объёма данных, добавление новых схем становится гораздо проще, чем создание новой базы данных. Это позволяет избежать сложностей с миграцией данных и синхронизацией между базами данных.

С другой стороны, если проект или система масштабируется до очень больших размеров (например, работа с большими объёмами данных или множеством параллельных пользователей), может возникнуть необходимость в распределении нагрузки. В таких случаях одна большая база данных с множеством схем может не справляться с высокой нагрузкой, и потребуется решение, которое может включать шардирование или использование нескольких баз данных.

Однако для большинства средних и крупных проектов использование одной базы данных с различными схемами будет достаточным для удовлетворения требований масштабируемости. Важно отметить, что PostgreSQL предоставляет механизмы, такие как индексы и партиционирование, которые позволяют оптимизировать работу с большими объемами данных даже в рамках одной базы данных.

## 3. Problem Statement и Как мы его Решаем

### 3.1 Исходная Задача и CustDev

- **CustDev по фреймворку Job to Be Done**  
  Показывает, что многие рестораны нуждаются в централизованной административной панели, сочетающей функции управления персоналом (HRM) и клиентами (CRM).  
- **Фокус на eNPS**  
  Важность метрики Employee Net Promoter Score, позволяющей быстро оценивать вовлечённость сотрудников и улучшать рабочие процессы.

### 3.2 Решение в Форме ZernoHub

1. **Универсальная CRM + HRM платформа**  
   Менеджеры видят все метрики (KPI, отзывы, eNPS) в одном месте.  
2. **Механизм регистрации и подтверждения**  
   Сотрудники получают доступ к системе только после одобрения администратором.  
3. **Модуль отзывов**  
   Клиенты могут оставлять комментарии и рейтинги, которые автоматически связываются с конкретным сотрудником или подразделением.

## 4. Архитектура: Repository → Service → Endpoint

В ZernoHub применяется многоуровневый паттерн разработки:

1. **Repository (Репозиторий)**  
   - CRUD-операции (Create, Read, Update, Delete).  
   - Прямое взаимодействие с базой данных PostgreSQL через ORM (например, SQLAlchemy).  
   - «Не знает» бизнес-логики, фокусируется только на чтении/записи.

2. **Service (Сервис)**  
   - Центральная точка бизнес-логики.  
   - Открывает транзакции (`session.begin()`) и вызывает методы репозиториев.  
   - При необходимости вызывает другие сервисы с параметром `nested=True` для вложенных транзакций.  
   - При успехе коммитит изменения, при ошибках откатывает.

3. **Endpoint (REST API)**  
   - Реализован на FastAPI (или другом фреймворке).  
   - Определяет HTTP-роуты для фронтенда или мобильных приложений.  
   - Вызывает соответствующие сервисы, обрабатывает и формирует ответы в формате JSON.

### 4.1 Преимущества

- **Разделение обязанностей**  
  Легче поддерживать, тестировать и наращивать функционал.  
- **Гибкое управление транзакциями**  
  Бизнес-логика решает, когда «зафиксировать» изменения.  
- **Масштабируемость**  
  Можно легко добавлять новые CRUD-методы и сервисы без нарушения общей структуры.

## 5. Бизнес-логика и Функциональность

### 5.1 eNPS (Employee Net Promoter Score)

- **Сбор данных**  
  Сотрудники периодически оставляют оценку (как сильно они готовы рекомендовать свою работу другим).  
- **Аналитика**  
  Менеджеры видят изменения во времени, выявляют проблемные зоны.  
- **Взаимосвязь с HRM**  
  Результаты eNPS могут влиять на пересмотр зарплат, поощрения и продвижения.

### 5.2 Feedback от Клиентов

1. **Формы / QR-коды**  
   Быстрый способ для гостей поделиться мнением.  
2. **Привязка к сотрудникам**  
   Если гость захотел указать официанта, платформа фиксирует связь «Отзыв → Сотрудник».  
3. **Просмотр и ответ**  
   Менеджеры могут отвечать на отзывы (если оставлены контакты), сортировать и анализировать статистику.

### 5.3 Авторизация и Регистрация

1. **Отправка запроса**  
   Новый пользователь (сотрудник/менеджер) регистрируется через специальную форму.  
2. **Подтверждение администратором**  
   После одобрения создаётся аккаунт с соответствующей ролью.  
3. **Управление профилями**  
   Пользователи могут видеть собственную статистику, менеджеры — аналитику по отделу.

## 7. Резюме и Потенциальные Направления Развития
Дополнительные Модули

Расширенный анализ данных (Data Science методы).
Автоматизация найма (подбор персонала, интеграция с job-порталами).
Безопасность

Расширенный контроль ролей и политик доступа (RBAC).
Валидация входящих данных (Pydantic, фронтенд-валидация).
Интеграции

Подключение чат-ботов (Telegram, WhatsApp) для уведомлений о новых отзывах.
Синхронизация с системами бронирования столиков и POS-системами.
Масштабирование

При увеличении базы клиентов и сотрудников возможно вынесение отдельных модулей (например, статистику и аналитику) в отдельные микросервисы.
CI/CD

Настройка конвейеров (pipeline) для автоматической сборки, тестирования и деплоя, что обеспечит стабильное развитие проекта.
## Заключение
ZernoHub — это многоуровневая система, которая сочетает в себе удобство интегрированной CRM & HRM-платформы и при этом остаётся гибкой для дальнейших улучшений. Благодаря модульной архитектуре (Repository → Service → Endpoint), PostgreSQL с использованием схем, а также разделению фронтенда (Evaluate для гостей и admin-panel-module для менеджеров) ресторан получает надёжное решение для управления персоналом и взаимодействия с клиентами.

При необходимости детально настроить бизнес-логику, разобраться в коде или внести свой вклад — изучите текущую документацию, открывайте дополнительные задачи (issues) в репозитории или свяжитесь с командой разработчиков. Постоянное совершенствование ZernoHub позволит ресторанному бизнесу повышать качество обслуживания и эффективно управлять важнейшими ресурсами — своими сотрудниками и клиентами.
